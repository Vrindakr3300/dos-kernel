**TL;DR** — DOS's loop spine had two of its three decision halves: `dos.health`
("should I start?") and `dos.loop_decide` ("should I continue?"). This release
adds the third — `dos.scout` ("**what** should I start?") — the pure chooser that
runs *before* a lane is leased and picks the next activity over the full menu
(dispatch / unstick / replan / wait / stop), instead of a host hardcoding
`iter-1 = dispatch` and only letting a post-acquire health gate insert a detour.
Scout is the **reference example of the dos composition idiom**: a pure
`choose(state)` that READS a sibling `HealthVerdict` as an input field — never
recomputes one — with all I/O kept on the host-adapter side of the wall. Operators
running fleets where a knowable-at-second-zero blocker (a doom-loop sweep, an open
operator decision, an overlap) keeps costing a `$2–9` child to rediscover are the
audience: scout reaches that decision at second zero, for `~$0`, before the lease.

## scout — the pre-dispatch chooser (`src/dos/scout.py`)

- **`choose(ScoutState) -> ScoutDecision`** — the canonical 9-rule decision order,
  first-match-wins, each rung carrying a `rule_id` so a test pins exactly which
  fired. Rules: (1) open escalated operator decision → STOP; (2) route_unstick +
  swept-≥2-today + operator-owned → STOP; (3) route_unstick + swept-≥3 +
  fixes-not-landing → STOP (the doom-loop); (4) route_unstick fresh → UNSTICK;
  (5) overlap → WAIT; (6) route_replan + open soak → WAIT; (7) route_replan /
  replan-due → REPLAN; (8) scoreboard zero-ship storm → STOP (`confidence: low`);
  (9) default → DISPATCH.
  - *Why:* the `job` repo's override log is ~30 transcripts of a human, before
    leasing a lane, reading the scoreboard / same-day-unstick-count / open-decision
    state and choosing stop-vs-replan-vs-unstick-vs-dispatch. Scout is that override
    made a pure function.
  - *How:* `dos.scout.choose`; the sibling `HealthVerdict` arrives via
    `ScoutState.health` (the `loop_decide.LoopState.liveness` seam) — scout reads it,
    never calls `dos.health.check` itself.
- **Pure by construction.** `choose()` does no I/O, no clock, no oracle call —
  every time-derived signal (`same_day_unstick_count`, `fixes_landing_since_unstick`,
  `open_soaks`, `replan_due`) is pre-reduced to a scalar/bool/tuple at the caller's
  edge (the `liveness.classify(now_ms)` discipline). An import-isolation invariant
  keeps `scout.py` free of `os`/`subprocess`/`pathlib`/`open`.
- **Throughline-first cut line.** Phase-1 live rules are 2/3/4/5/7/9 + the two named
  new signals; rules 1/6/8 and the wider menu (`next-up`/`judge`/`plan-audit`/
  `tail-wag`/`trajectory-audit`) ship in the schema but stay **dormant** via
  empty/None defaults and a bounded `_phase2_menu()` annex — so widening the menu is
  a zero-schema-break change.
- **39 replay tests** (`tests/test_scout.py`) — one class per rule, the override-log
  acceptance set, the `__post_init__` guards (no gate on a non-DISPATCH decision, no
  `cause_key` on a non-STOP/WAIT decision), and the CLI exit codes.

## cli — `dos scout` (`src/dos/cli.py`)

- **`dos scout --state-json '{…}'`** — mirrors `dos health`: a fixture-driven CLI
  that prints the `ScoutDecision` JSON and exits with the activity's code
  (0 dispatch · 3 unstick · 4 replan · 5 wait · 6 stop), flipping any
  `confidence: low` pick to exit 6 (surface, never auto-run — the `JO_AUTO_ACCEPT`
  boundary made visible to a shell caller).
  - *How:* `cmd_scout` → `scout.cmd_check`; no `__init__` re-export (the
    `health`/`loop_decide` direct-import pattern).
