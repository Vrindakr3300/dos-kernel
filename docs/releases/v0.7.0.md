**TL;DR** — `verify`'s evidence grade is now visible: at a TTY, `dos verify` colors
the `SHIPPED`/`NOT_SHIPPED` mark and the `(via <rung>)` provenance suffix, and
`dos doctor` opens with one honest line saying whether DOS can verify this repo's
claims at all. Both are CLI-boundary presentation — the byte-faithful `text`/`json`
renderers and every machine/pipe consumer are unchanged. This release also closes the
lane-journal write side (the WAL that makes `liveness` SPINNING reachable from real
evidence) and hardens atomic file replacement on Windows. Operators running fleets and
anyone wiring `dos verify` into CI both benefit.

## `verify` — the provenance rung is now the tell (`oracle` / `cli`)

- **`dos verify` colors the verdict for a human at a TTY.** A confirmed ship
  (`via grep` / `via registry`) reads green; the no-evidence `(via none)` reads
  bold-red — the self-incriminating "the agent said done, git proved nothing" tell,
  in one glance.
  - *Why:* the evidence-grade rung is `verify`'s most-differentiated idea; it used to
    be gray parenthetical wallpaper. Making it the screenshot's signal is the point.
  - *How:* `cli._color_verdict_line` / `_color_enabled`, applied at the CLI boundary —
    **never** in the renderer, so the RND byte-faithfulness contract
    (`tests/test_render.py`) is untouched. Honors `NO_COLOR` (which beats
    `DOS_COLOR=always`, per no-color.org) and `DOS_COLOR=always|never`.
  - *Contract:* a pipe, a redirect, `--json`, and any plugin renderer all receive the
    exact bytes — `dos verify | grep 'via none'` and CI logs are unchanged, and the
    exit code (`0` shipped / `1` not) is never touched by color. Pinned by
    `tests/test_verify_color.py`.

## `dos doctor` — the always-honest verifiability cold-open (`cli` / `config`)

- **`dos doctor` now opens with one line, correct on every repo, saying whether
  `dos verify` can check this repo's claims at all.** "verifiable now: N of your last
  50 commits name a unit of work `dos verify` can check," or — on a repo that stamps
  nothing — "none of your last M commits name a unit of work — no referee can check
  agent claims yet."
  - *Why:* the on-ramp that is true on *every* repo, including the Conventional-Commits
    majority where `verify` is otherwise silent. It tells a stranger in five seconds
    whether their repo even speaks the grammar a referee needs.
  - *How:* `cli._verifiability_headline` (text) + a `verifiability` block in
    `dos doctor --json` (`commits_read` / `ship_shaped` / `recognized` / `grammar`),
    computed from the repo's own recent subjects via the same two pure stamp
    predicates the grep rung uses (`stamp.ship_shaped_under_generic` +
    `StampConvention.recognizes_direct_ship`) — no second, drifting heuristic.
  - *Contract:* it **never cries wolf** — a cold repo gets the honest "no referee can
    check" form, never a false `via none` accusation. Pinned by `tests/test_stamp_doctor.py`.

## Lane-journal — the WAL write side closed (`lane-journal`)

- **The lane journal now records the events it could only previously read.** A
  `HEARTBEAT` beat (which makes `liveness` SPINNING reachable from *real* evidence, not
  just absence), `REFUSE` records for denials, and an `OP_CHECKPOINT` compaction
  (`dos journal compact`).
  - *How:* `src/dos/lane_journal.py` + `src/dos/lane_lease.py` + the CLI verbs; the
    pure fold (`journal_delta`) is unchanged. Pinned by `tests/test_lane_journal.py`,
    `tests/test_lane_lease.py`, `tests/test_journal_delta.py`.

## Kernel hardening (`config` / spine)

- **win32-hardened `atomic_replace`**, routed under `run_id` and `home` writes — the
  `os.replace` race on a case-insensitive/locked filesystem the Windows platform hits.
  *How:* `src/dos/_filelock.py`; pinned by `tests/test_filelock.py`.
- **MCP server `{workspace}` resource segment is percent-encoded** (fixes a Linux CI
  failure). *How:* `src/dos_mcp/server.py`; pinned by `tests/test_mcp_server.py`.
