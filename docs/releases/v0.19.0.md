**TL;DR** — Three landings since v0.18.1. Two new features make the fleet more
observable: `dos top` now surfaces a run that has *launched but not yet leased* a
lane (the SPAWN→ACQUIRE blind window) as a `SPAWNING` chip, and a new advisory
`Stop` hook keeps uncommitted/unpushed work visible at every turn-end without ever
blocking. One real fix retires a serialization regression v0.18.1 shipped: a
concurrent acquirer could double-book a region whose holder process had already
exited. Anyone relying on `dos arbitrate` / `dos lease-lane acquire` to serialize
concurrent writers should take the fix; the two features are additive and
opt-in-by-observation.

## `dispatch-top` — make a spawning loop visible (feat)

- **`dos top` now shows a `🔵 SPAWNING` chip** for a lane a run is *coming to* but
  has not yet leased — the window between a launcher committing to a lane and its
  `ACQUIRE` landing in the WAL.
  - *Why:* `arbitrate` persists nothing on the spawn path, so a loop that had
    launched but not yet acquired was invisible in `dos top` — it looked idle when a
    run was inbound (the 2026-06-09 dos-top spawning-loops operator audit, private archive).
  - *How:* `src/dos/dispatch_top.py` folds recent `OP_SPAWN` entries for any lane
    with no live lease into a `SPAWNING` chip on a short TTL; a held lease always
    wins the chip, so an eventual `ACQUIRE` supersedes the spawn and a stale spawn
    ages out on its own rather than wedging a phantom `SPAWNING`.

## `lane-journal` — a non-mutating intent-to-spawn op (feat)

- **New `OP_SPAWN` journal op** records "a run is coming to this lane" the instant a
  launcher commits — the acquire-side symmetric intent.
  - *Why:* it closes the `SPAWN→ACQUIRE` blind window the `dos top` audit found.
  - *How:* `src/dos/lane_journal.py:spawn_entry` / `OP_SPAWN`. The op is *non-mutating*
    — like `HALT`/`ATTEMPT`, it grants no lease and removes none; the structural
    replay fold ignores it for admission and reads it only for the `dos top`
    `SPAWNING` chip. `src/dos/cli.py` and `src/dos/lane_lease.py` carry the spawn
    plumbing.

## `tooling` — an advisory git-hygiene Stop hook (feat)

- **A second `Stop` hook now nudges when the tree carries stranded work.** It prints
  a one-line *"N uncommitted files on master, K commits not pushed — commit the lane
  or run /release"* whenever the working tree is dirty or the branch is ahead of its
  upstream.
  - *Why:* keep "ship the finished work" the default reflex, so a session never
    quietly ends leaving the tree dirty and the work unshipped — the lightweight
    always-on complement to the deliberate `/release` skill.
  - *How:* `scripts/git_hygiene.py`, wired in `.claude/settings.json`. **Advisory by
    construction (the docs/274 rule):** it prints to stderr and always exits 0 in its
    default mode — it never blocks a turn, commits, pushes, or removes anything (a
    blocking `Stop` hook would force-loop interactive sessions). The one non-zero exit
    is the explicit opt-in `--strict` / `DOS_GIT_HYGIENE=strict` mode for a headless
    loop.
  - *How (lease-aware):* it folds DOS's own kernel WAL (`dos.lane_journal`, the same
    fold `dos top` and `/release` Step 1.6 read) and subtracts paths a live lane
    lease legitimately owns mid-flight, so the nudge fires on the *stranded* count,
    not the raw dirty count. A `chore` commit also wires a throttled Claude-state
    cleanup into the same hook.

## `lane-lease` — acquire reads the structural fold, not dead-elision (fix)

- **`acquire()`'s contention read reverts** from `live_leases(config, expire_dead=True)`
  to `live_leases(config)`, ending a serialization regression v0.18.1 introduced.
  - *Why:* a lane lease can be held by a process that legitimately exits between
    `ACQUIRE` and `RELEASE` — its *effect* (the booked region) outlives the
    short-lived `dos lease-lane acquire` child. The dead-PID rung probed that
    already-exited PID, judged the still-held region dead, elided it, and let a racing
    acquirer book the same region: a lost update — the exact TOCTOU the lease exists
    to prevent.
  - *How:* `src/dos/lane_lease.py:acquire`, inside the `_Mutex` where acquirers are
    already serialized, so dead-elision is neither needed nor safe.
  - *Impact:* anyone relying on `dos arbitrate` / `dos lease-lane acquire` to
    serialize concurrent writers should take this fix; it restores deterministic
    serialization (the worked example's K=4 coord demo is green again).
- **What did NOT change** — the docs/281 phantom-orphan self-heal. The long-lived
  admission hook keeps `expire_dead=True`, so a crashed worker's orphaned lease still
  stops revoking a live interactive session's tools. There were exactly two
  `expire_dead=True` call sites; only `acquire()`'s was wrong.

## `hook` — phantom-lease Go/Python parity

- **The Go `dos-hook` twin gains WAL-expiry tests and the parity corpus is
  regenerated**, completing the docs/281 phantom-lease parity across the Python and
  Go fast-paths.
  - *How:* `go/internal/hook/wal_test.go`, `go/internal/hook/parity/{corpus.jsonl,gen_corpus.py}`,
    plus `tests/conftest.py` / `tests/test_hook_pretool.py` coverage.

## Provenance

The acquire fix was found by auditing a "the substrate is sound" claim against the
suite rather than trusting it (`docs/283`): `test_coord_demo_k4_serializes_writes`
was red on the v0.18.1 mainline. The regression and its mechanism were proven with a
stash-only A/B; the fix restores deterministic green across the coord demo, lease,
arbiter, admission, and hook suites. The `dos top` visibility gap was found and fixed
in the 2026-06-09 dos-top spawning-loops operator audit (private archive).
