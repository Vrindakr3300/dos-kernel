# The DOS build journal — design notes, plan records, research arcs

The *why* behind each decision and the phased plans that shipped (or are still
on the bench). Nothing here is required to *use* DOS — the guides live in the
**[docs index](README.md)** — but this is the best way to understand the
kernel, and the right thing to read before changing it.

The docs are numbered in the order they were written (66 → 271); the number is
chronology, **not** a reading order. (A handful of early numbers —
82/84/99/117/139/156 — were written twice; the later-renumbered halves now live
at **182–187**, listed in the research-arc tables below. Three collisions are
still open, so a bare "docs/NN" reference to any of these is ambiguous until
one of each pair is renumbered: **191** —
`191_proactive-dos-banking-the-verdict-before-the-failure` vs
`191_tool-call-division-and-the-pretool-hook`; **125** — the
`125_go-hook-fastpath-build-plan` (the Go native fast-path, indexed below) vs the

`267_durable-release-concepts-from-the-userland-app` vs
`267_the-webhook-notifier-the-universal-transport-on-the-notify-seam`. Each pair's
two halves are distinct documents that happen to share a chronology number; cite
them by full filename, not the bare number.)

## Design notes — the *why* (read for understanding)

These are essays, not plans. They explain the thinking the code rests on; none of
them is required to *use* DOS, but they're the best way to understand it.

| Doc | The idea |
|---|---|
| [79_primitives-not-features](79_primitives-not-features.md) | Why the syscalls are deliberately small — a primitive you build *on*, not a feature you consume. The restraint that makes a substrate. |
| [182_the-kernel-is-a-taxonomy-of-refusal](182_the-kernel-is-a-taxonomy-of-refusal.md) | All four syscalls are kinds of "no" — disbelief, decline, deny-effect, deny-anonymity. Refusal is the load-bearing primitive. |
| [102_when-to-trust-an-agent](102_when-to-trust-an-agent.md) | The other half of the slogan: *when is it reasonable to trust an agent?* Trust structure not content (the Linux `copy_from_user` posture), prior commitments not reports (a plan is a predicate — it turns undecidable "good work?" into decidable "conforms?"), and only where a wrong yes is detectable + reversible (which *derives* the ORACLE→JUDGE→HUMAN layering). Frames each core problem against its next-best alternative. |
| [103_memory-is-an-unverified-agent](103_memory-is-an-unverified-agent.md) | Applies 102's trust law to the agent's *own persistent memory*: a memory file is a frozen self-report, and recall injects it as fact. Reproduced live on this repo's store (a "FIXED cli.py" claim that git says shipped two commits ago). Every memory failure is a syscall re-aimed — stale "FIXED" is `verify`, aging facts is `liveness`, confidently-wrong recall is the missing `refuse`/abstain rung, index bloat is `arbitrate` over topics. The fix is a recall *driver* (a consumer, not a kernel change): re-verify a memory's checkable claims against ground truth at read, surface a closed `RECALL_FRESH/DRIFTING/STALE/UNVERIFIABLE` verdict, never the bare claim. |
| [108_the-cheap-lie-and-the-narration-taxonomy](108_the-cheap-lie-and-the-narration-taxonomy.md) | The realistic fleet failure isn't the *fabricated diff* (rare, expensive, shape-checkable) — it's the cheap, confident one-liner *"all work completed"* a worker emits over an empty tree: common, costless, eye-undetectable, and the line `verify()` is built to ignore. Lays out the four-way narration taxonomy — fake-edit and the cheap fake-narration both bottom out on `verify` reading git (the flake is indistinguishable from the lie *without* it); intent-mismatch and context-collapse escape the oracle to the JUDGE/HUMAN rung. Follows the regress down: the judge is itself a narrator (contained, advisory-only, not made correct), and it terminates only at the non-narrating, forgery-proof git floor. |
| [76_flexible-goals-and-verification](76_flexible-goals-and-verification.md) | Where the system *bends* (which signals, which provenance) and where it stays *rigid* (the adjudication itself). |
| [80_mcp-server-surface](80_mcp-server-surface.md) | DOS as a tool any agent can call — the MCP server as the lowest-friction adoption door. |
| [81_velocity-economics-and-the-fleet-benchmark](81_velocity-economics-and-the-fleet-benchmark.md) | The second axis of fleet effectiveness: catching the lie is integrity, *not paying for it downstream* is velocity. Defines the benchmark category. |
| [98_the-orchestrator-is-a-driver](98_the-orchestrator-is-a-driver.md) | The orchestrator (DOS-dispatch vs an external harness/ultracode `Workflow`) is just another driver of the same trust seam — comparable on one workload. Ships `dos lease-lane` (durable lease write-back over the pure arbiter) so a cross-process flow keeps collision-prevention. |
| [99_runtime-validation-and-the-actuation-boundary](99_runtime-validation-and-the-actuation-boundary.md) | The three moments of a run (before / *during* / after); the historical runtime failures are "alive-but-not-moving", already detected by `liveness()` but consumed by nothing. Closes the loop (`StopReason.SPINNING` self-stop) and draws the actuation line: the kernel may not kill because it must not know *what a process is* — self-stop is control flow (in-bounds); a foreign stop is *record + propose* only (the `reap`-family `halt` verb). |
| [84_ground-truth-trajectories-for-training](84_ground-truth-trajectories-for-training.md) | The kernel as a training-data instrument — every step emits a verified (claim, evidence, verdict) tuple. |

- [217_the-cross-vendor-hook-dialect-seam](217_the-cross-vendor-hook-dialect-seam.md) — DESIGN. DOS has **three** binding surfaces: the kernel syscalls (vendor-*irrelevant* — they adjudicate git, not the agent), the **MCP server** (works on all three TODAY — all are MCP clients — but advisory: a tool the agent *calls*, not a gate), and the **hooks/PEP** (`dos hook pretool`/`posttool`/`stop`), which emit **exclusively the Claude Code `hookSpecificOutput` envelope**. Cheap because the vendors copied CC's payload shape — the deltas are event-names + envelope, not logic. Phase 1 gate = the CC dialect reproduces today's bytes exactly (existing hook suite green, unchanged).
