# 123 — The local model and the independence coordinate: where the judge runs is a trust fact, not a deployment detail

> **The kernel's entire relationship to a model is one shell-out: `drivers/llm_judge._call_provider` does `subprocess.run("claude -p …", shell=True)` — fork+exec an opaque CLI, read stdout. That seam silently assumes the model is *remote*: a service off-box whose bytes the judged agent cannot author. Every property of that assumption inverts when the model runs on the *same host* as the worker. A co-resident local model is not "a cheaper cloud model" — it is a different *trust object*, simultaneously MORE dangerous (the judge's generator may be the worker's generator — a model grading its own homework) and MORE honest-able (a local model has a weights digest + quant + sampler params: a content-addressed identity a cloud API never exposes). This note's claim is that "where the model runs" is not a deployment choice the kernel can ignore — it is a measurable coordinate in the *independence* of a JUDGE-rung verdict, and the witness-≠-actor law ([`117`](185_native-log-adapters-and-the-actor-witness-split.md)) already governs it. Cross-locality validation — a local model auditing a remote one, or a remote model adjudicating a local one's residue — is the *deliberate maximization* of that coordinate. The fix is the same seam move the kernel already made four times (judges, overlap, log-sources, evidence): lift an assumption (here: "the judge is independent of the worker") from *taken for granted* to *measured, declared, and floored so a swapped component can only refuse-MORE*.**

Status: theory + spec note, in the family of [`86`](86_the-typed-verdict-surface.md)/[`87`](87_the-adjudicator-trust-ladder.md) (the JUDGE rung), [`115`](115_the-environment-print-and-under-what-axis.md) (EnvPrint — *under what* a verdict ran), [`117`](185_native-log-adapters-and-the-actor-witness-split.md) (witness ≠ actor), and the device pair [`121`](121_first-class-on-devices-and-unattended.md)/[`122`](122_the-core-go-runtime-and-the-on-device-kernel.md). §1 separates this note from 121/122 (they put the *referee* on a small box; this puts the *model* on the same box as the worker). §2 is the core reframe: locality is a coordinate of independence, and the seam mis-models it both ways. §3 is the 2×2 of (worker locality) × (judge locality) and why the two *cross* cells are the interesting ones — the cross-locality validation the prompt asks about. §4 is the mechanism (the four threads: transport shape, independence-as-measured, identity-stamping, accelerator contention) and how each is an existing seam re-aimed. §5 is the buildable spec. §6 steelmans. §7 is the litmus set. Nothing here is built.

The positioning half — *why a trust kernel that can referee a local model is a market wedge (private/offline/cheap adjudication, sovereignty over the referee)* — is a [`dos-private`](../../dos-private) concern and is deliberately not argued here (CLAUDE.md: how-a-module-behaves → `dos/docs`; why-it-matters → `dos-private`). This is the mechanism half.

---

## 1. This is not 121/122 — those move the *referee*; this moves the *model*

[`121`](121_first-class-on-devices-and-unattended.md) and [`122`](122_the-core-go-runtime-and-the-on-device-kernel.md) ask: *can the DOS kernel — the referee — run on a phone / robot / MCU?* Their answer is the `EvidenceSource`/`DurableLog` seams (121) and the static Go core (122). That is a question about the **topology of the adjudicator**.

This note asks a different question that 121/122 never touch: *what changes when the **model** — the thing being refereed, and/or the model doing the refereeing — runs on the **same host** as the kernel and the worker?* That is a question about the **topology of the generators**, and it is orthogonal to where the kernel runs:

| | 121 / 122 | This note (123) |
|---|---|---|
| **What moves to the edge** | the kernel (the referee) | the model (the worker and/or the judge) |
| **The hard primitive** | `EvidenceSource` / `DurableLog` (ground truth + durability without a capable host) | the JUDGE rung's **independence** (a co-resident judge may share the worker's generator) |
| **The governing law it leans on** | 117's witness ≠ actor, for *evidence* | 117's witness ≠ actor, for the *judge* |
| **The seam shape it re-uses** | overlap's `admissible_under_floor` | judges' fail-to-abstain + EnvPrint's content-addressed identity |

They compose — a 122 on-device kernel will *often* be adjudicating a co-resident local model, so the two questions meet at the edge — but they are distinct, and conflating them hides this note's actual content. **121/122 make the referee deployable; 123 makes the referee honest about a model it shares a box with.** The rest of this note is the second problem.

## 2. The reframe: locality is a coordinate of *independence*, and the seam mis-models it both ways

Read `drivers/llm_judge.py` and the kernel's *whole* coupling to a model is one function:

```python
# llm_judge.py:88 — the ENTIRE interconnection between DOS and any model
def _call_provider(prompt: str) -> str | None:
    cmd = os.environ.get("DOS_LLM_JUDGE_CMD")        # e.g. "claude -p --model …"
    p = subprocess.run(cmd, shell=True, input=prompt.encode(), …)
    return p.stdout.decode().strip() or None
```

Fork a shell, exec an opaque CLI, read a line of stdout. This seam encodes an **unstated assumption**: that the model is *remote* — a cloud service reached through a CLI, whose bytes are authored somewhere the judged agent cannot reach. Under that assumption the JUDGE rung's independence is *free*: the `claude -p` judge runs in Anthropic's datacenter, the worker runs on the operator's box, and there is an air-gap of accountability between them. The seam never has to *check* independence because the deployment shape guarantees it.

Every clause of that assumption breaks when the model is local and co-resident:

| Property | What the seam assumes (remote) | What a co-resident local model is |
|---|---|---|
| **Independence (the load-bearing one)** | judge's bytes authored off-box; the worker cannot influence them → independence is structural and free | judge and worker on the same host, possibly the **same process tree**, possibly the **same weights** (one Ollama instance serving both) → independence is *not* guaranteed and must be *measured* |
| **Invocation** | fork+exec a CLI per call (`shell=True`) | a long-lived local server you socket to (Ollama `:11434`, a llama.cpp server, vLLM) — or an in-process FFI call (llama.cpp via ctypes, MLX) where there is **no fork at all** |
| **Cost** | network latency; someone else's compute and energy | local accelerator joules, drawn from the *same* GPU/NPU/thermal budget the worker is using (the 122 §3 energy axis, now *contended*) |
| **Identity** | unversioned, changes under you, unhashable (the model silently upgrades) | a weights file (a content hash), a quantization, a seed, a temperature, a context length — a **content-addressed, stampable identity** (the [`115`](115_the-environment-print-and-under-what-axis.md) EnvPrint shape, which already flagged "a pinned weight set is not a pinned behavior" — it *anticipated* this object) |

The deep observation: these are not four unrelated wrinkles. **They are all aspects of one quantity the seam currently cannot express — how independent the judge's generator is from the worker's — and locality is its cheapest, most observable proxy.** The witness-≠-actor law ([`117`](185_native-log-adapters-and-the-actor-witness-split.md)) says *a judgment is evidence only when its byte-author is not the judged agent.* Generalize "byte-author" to "generator." Then:

> **The JUDGE rung's whole worth rests on the judge's generator being distinguishable from the worker's generator. Locality (which host? which weights? which provider?) is a *measurable component* of that distinguishability. A cloud judge over a CLI maximizes it by accident; a co-resident local model can collapse it to zero (same weights → the model grades its own homework → generation #2 about generation #1 from the *same* generator, which is exactly the canonical *non*-witness). The seam is built for the accidental-max case and is silent — dangerously silent — in the collapse case.**

That silence is the bug. `judges.py` has four disciplines (deterministic-first, advisory-only, fail-to-abstain, abstention-first) and **every one defends against a judge that is *wrong*. None defends against a judge that is the *same generator* as the worker** — a judge that is confidently, independently-looking, *wrong in the worker's exact direction* because it shares the worker's blind spots. A `Claim` carries `{claim_text, stated_reason, evidence, subject}`; a `JudgeVerdict` carries `{stance, why, evidence, cost}`. **Nowhere is there a field for who the judge is, or how far its generator sits from the worker's.** The kernel cannot today tell "an independent model cleared this" from "the worker cleared its own claim wearing a judge hat."

## 3. The 2×2: cross-locality validation is the deliberate maximization of the independence coordinate

The prompt's sharpest addition — *use a local model to validate a remote model, or vice versa* — is exactly the move that makes the independence coordinate *large on purpose*. Lay out (worker locality) × (judge locality):

| | **Remote worker** | **Local worker** |
|---|---|---|
| **Remote judge** | both cloud. Independence depends entirely on *different model / provider* — same trust domain, no locality gap to lean on. (Today's `claude -p` judging a `claude` worker is here, and is *weaker* than it looks if the model is the same.) | **cross ↑** — local worker, remote judge. The judge is off-box; the worker cannot touch the judge's weights, context, or compute. **Strong** independence on the locality axis. The *escalation* cross. |
| **Local judge** | **cross ↓** — remote worker, local judge. Your own pinned local model audits a cloud worker's claims. Strong independence (different box, different *control domain*) **and** the referee is the one you own. The *sovereignty* cross. | both local — **the danger cell**. Same box, maybe same weights, same accelerator. Independence is *least* guaranteed and contention is *highest*. Worst case: literally one Ollama instance grading itself. |

The two **cross** cells are the ones the prompt is pointing at, and they are **not symmetric** — they succeed for *different reasons*, which is why naming both matters:

### 3.1 Local judge / remote worker (cross ↓) — *sovereignty over the referee*
You are renting an expensive, uncontrollable cloud model to do the work. You cannot pin it, cannot see its weights, cannot run it offline, and it changes under you. But you can run a *cheap, pinned, private, offline-capable* local model as the **judge** — and audit the cloud worker's claims with a referee *you* control. The independence is strong (different host, different control domain) *and* the operator owns the adjudicator. This is the **economically natural** cross: spend cloud tokens on the work, spend local joules on the *check*. "Trust-but-verify the API you are renting." Its value is **control of the referee**, and it is the natural default for an operator who distrusts a vendor's self-grading.

### 3.2 Remote judge / local worker (cross ↑) — *escalation to an independent referee*
Your worker is a local model — cheap, private, fast, but limited. For the claims that *matter* (the ones the deterministic oracle abstained on, and a co-resident local judge cannot independently clear — §3's danger cell), you escalate the **judgment** to a remote model the local worker provably could not have influenced: different weights, off-box, the worker never saw the judge's context. This is the **trust-ladder** ([`87`](87_the-adjudicator-trust-ladder.md)) made multi-rung *within* JUDGE: deterministic oracle → local judge → **remote judge** → human, each rung up *more independent and more expensive*. Its value is **independence on demand** — you pay for the distant referee only on the residue that needs it.

### 3.3 The unification — locality is a rung, and the danger cell is also the contended cell
The 2×2 is not four deployment options; it is a **gradient of independence** the kernel should *measure and route on*:

1. **Independence is rankable, and locality difference is its cheap proxy.** A judge on a different host / different weights / different provider than the worker is *more* of a witness than one co-resident with it. The kernel can *observe* each of those differences (compare producer prints — §4.2) and order them. Cross-locality is just "make the difference maximal."
2. **The floor discipline carries verbatim.** A judge that *cannot be shown* independent of the worker can only ABSTAIN-more, never AGREE-more — the overlap-floor / fail-to-abstain pattern re-aimed at independence. Same-box-same-weights → the kernel structurally refuses to let that verdict *clear* a claim; it degrades to "no independent witness present → escalate / ask a human." A buggy or hostile "independence scorer" can only make the verdict *more* conservative, exactly as `admissible_under_floor` lets an overlap policy only refuse-more.
3. **The elegant coincidence: the least-independent cell is also the most-contended cell.** In the local/local danger cell the judge and worker (a) fail independence *and* (b) fight for the same accelerator (§4.4). **Both have the same root cause — co-residence — and the same fix: push one side off-box.** Cross-locality validation is therefore simultaneously the *independence* fix and the *contention* fix. You do not trade one for the other; the move that buys an independent referee also frees the GPU. That is the strongest reason the cross cells are not a niche — they are where both problems dissolve at once.

## 4. The mechanism — four threads, each an existing seam re-aimed

Nothing here is a new subsystem. Each thread is a seam the kernel already shipped, pointed at the inference engine on the same host. (This is the 121 move — *"the fix is the same seam move the kernel already made three times"* — a fourth and fifth time.)

### 4.1 Transport shape — a `ModelTransport` seam (the 122 split, re-aimed at the judge)
The `_call_provider` shell-out is one transport — *fork+exec a CLI* — and it is the wrong one for a co-resident server. [`122 §5.2`](122_the-core-go-runtime-and-the-on-device-kernel.md) already named *"two invocation shapes, one core"* for the Go verdict cores (subprocess vs in-process FFI). The same split applies to the judge:

- **`cli`** — today's `subprocess.run(shell=True)`; the cloud-CLI floor, kept verbatim.
- **`socket`** — an HTTP/IPC call to a long-lived local server (Ollama `:11434`, a llama.cpp server, vLLM, an OpenAI-compatible local endpoint). The *natural* shape for a local engine — no per-call fork, a warm model.
- **`inproc`** — an in-process FFI call (llama.cpp via ctypes, MLX in-process). No fork at all — the 122 `c-archive`/FFI shape, here for the *model* rather than the verdict core.

The transport is a property of the *driver*, declared in `dos.toml`, never in the kernel — the existing kernel/driver split (`judges.py`: "every real judge with provider surface lives in a driver"). The kernel gains nothing; a new `OllamaJudge` / `LocalSocketJudge` driver picks its transport. **Why this matters for trust, not just ergonomics:** a `socket`/`inproc` transport makes "where the model runs" *observable to the kernel* — a localhost endpoint and a loaded weights file are facts the kernel can read, which is precisely what lets locality become a *measured* coordinate (§4.2) instead of an opaque CLI string.

### 4.2 Independence as a measured fact — `EnvPrint` of the *producer*, compared
This is the load-bearing thread. The kernel already has the object that identifies *under what* a verdict ran: [`115`](115_the-environment-print-and-under-what-axis.md)'s `EnvPrint` — a content-addressed `{kernel_version, kernel_sha, python, platform, tools}` with a stable `digest` (the `EnvId`). And it *already anticipated the model*: `env_print.py:30` records the caveat *"a pinned weight set is not a pinned behavior."* The `tools` set — `ToolVersion{name, version}`, declared in `dos.toml [env]` — is the exact slot a model's identity belongs in:

```
ollama:qwen2.5-coder:7b-instruct-q4_K_M @ sha256:abc…  temp=0  seed=0  ctx=8192
```

is a declared "tool" whose "version" is its digest. So the mechanism is:

1. **Stamp each generator with a producer `EnvPrint`** that *includes its model identity* — the worker's print and the judge's print. (A worker that is a local model declares its model in `[env]`; a judge driver reports the model it called.)
2. **Derive independence by comparing the two prints.** Same `digest` → same generator → independence is *zero*. Same `platform`/host but different weights → independence is *partial*. Different host *and* different weights *and* different provider → independence is *high*. The comparison is exactly the [`115 §5`](115_the-environment-print-and-under-what-axis.md) `FLEET_ENV_MISMATCH` machinery — **which is specced but not yet built** (`lane_journal.py:454`: *"the gate that COMPARES a digest to a pin is a later phase"*). This note gives that pending gate its second consumer: it compares not "worker vs a declared pin" but "judge vs worker," and routes on the distance.
3. **Floor it.** Carry the derived independence onto the `JudgeVerdict` (a new optional field — the `cost` precedent: optional, defaults to a safe zero). A verdict whose judge cannot be shown independent of the worker is **structurally barred from AGREE** — it degrades to ABSTAIN ("no independent witness → escalate"), the fail-to-abstain discipline (`judges.py:243`) extended from "the judge failed" to "the judge was not independent." A `same-weights` self-judgment can no more clear a claim than a raising judge can.

This is the whole answer to the danger cell, and it is *additive* — a workspace that declares no model identity gets `independence = unknown`, which (conservatively) cannot AGREE on anything that needs independence, exactly as a missing `EnvPrint` reads as "not recorded."

### 4.3 Identity-stamping — the one place a non-deterministic rung gets reproducible provenance
A corollary of §4.2 that is worth stating on its own, because it is the *positive* inversion. A cloud judge's verdict is unreproducible: the model is unversioned and non-deterministic, which is *the reason* the JUDGE rung was exiled from the kernel in the first place ([`86`](86_the-typed-verdict-surface.md): the kernel is deterministic; judges are not, so they live in drivers). A **local** judge breaks that one-way door a crack open: with a pinned weights digest, `temp=0`, and a fixed seed, a local judge's verdict becomes *re-runnable* — stamp the verdict with the producer `EnvPrint` and another operator can recompute it under the recorded print and see if it holds. It is still not a *behavioral* guarantee (115's caveat stands: pinned weights ≠ pinned behavior; batching/kernel/hardware non-determinism remains), but it is **strictly more contestable than a cloud verdict**, which cannot be re-run at all. The local model is the first JUDGE-rung occupant whose ruling carries a content-addressed, re-executable provenance — the EnvPrint stamp, now naming the model, turns an advisory opinion into an *audit trail with a reproduction recipe*.

### 4.4 Accelerator contention — a symptom, not a separate problem
In the local/local cell, the judge and worker contend for one GPU/NPU. DOS has a resource-arbitration primitive — `arbitrate`/lanes ([`89`](89_the-lane-is-a-region-lock.md)) — but it arbitrates *file regions*, not *compute*, and [`77`](77_the-hardware-resource-manager.md) (HRM, the `place()` verb) is the doc that would extend it to accelerators. This note does **not** propose building accelerator arbitration; it observes that **§3.3's coincidence makes it largely unnecessary for the trust story.** Contention is a *symptom* of low independence: the cell where judge and worker fight for the GPU is the same cell where the verdict cannot be trusted anyway. The independence floor (§4.2) already steers you *out* of that cell for any claim that matters — and out of the cell, the contention is gone (the judge is off-box, or the worker is). So the trust-correct configuration and the resource-correct configuration are the *same* configuration. Where an operator *does* deliberately keep both local (cost/privacy reasons, on claims that do not need a distant referee), accelerator scheduling is a host concern (a serialized queue in the driver), not a kernel verdict — the [`99`](99_runtime-validation-and-the-actuation-boundary.md) actuation boundary: the kernel records and refuses, it does not schedule the GPU.

## 5. The buildable spec

The throughline slice is §4.2 (independence as a measured fact), because it is what makes *every* local-model configuration safe and it reuses the exact `EnvPrint` + fail-to-abstain apparatus. Pure-kernel; the model drivers are out-of-kernel.

- **`Claim` / `JudgeVerdict` gain an independence handle (kernel, additive).** A `Claim` optionally carries the *worker's* producer `EnvPrint` (who made the claim). A `JudgeVerdict` optionally carries the *judge's* producer `EnvPrint` (who ruled) and a derived `independence` (a small enum: `INDEPENDENT` / `SAME_HOST` / `SAME_GENERATOR` / `UNKNOWN`). Both default to absent/`UNKNOWN` — a workspace that stamps nothing behaves exactly as today, the `cost`-field precedent (optional, safe zero).
- **The independence floor (kernel, pure — the load-bearing rule).** A composition helper (`judge_eval.compose_deterministic_first`'s sibling) AND-s the floor: **a verdict may AGREE only if its judge is shown `INDEPENDENT` of the worker.** `SAME_GENERATOR` / `SAME_HOST` / `UNKNOWN` → the AGREE is downgraded to ABSTAIN with a typed reason ("no independent witness — escalate"). The dual of `admissible_under_floor`: a swapped independence scorer can only make a verdict *more* conservative. This is the §3.3(2) discipline, in code.
- **The comparison reuses the pending `FLEET_ENV_MISMATCH` gate ([`115 §5`](115_the-environment-print-and-under-what-axis.md)).** That gate already compares an `EnvPrint.digest` to a declared pin by equality; this note's comparison is judge-print vs worker-print, with a coarsening (same digest / same `platform` / different) into the `independence` enum. Building this gives 115 §5 its concrete second consumer and motivates finally landing it.
- **`drivers/` — the local-model judges (out of kernel).** An `OllamaJudge` (`socket` transport to `:11434`, reports the served model's digest), a `LocalSocketJudge` (any OpenAI-compatible local endpoint), an `InProcJudge` (llama.cpp/MLX FFI). Each is a `dos.judges` occupant exactly like `LlmJudge`; each reports its model's producer print so §4.2 can compare. The kernel imports none of them — the existing `no dos.drivers import` litmus covers it across the new transports.
- **`dos doctor` surfaces the independence story.** Alongside the active judges, doctor reports, per configured judge, its model identity and the `independence` it would have *against the declared worker model* — so an operator can *see* "your local judge shares the worker's weights; it cannot clear anything; wire a remote judge for the residue." The 121 honesty-floor-is-visible discipline, applied to the judge.

The cross-locality cases (§3.1/§3.2) then fall out as *configuration*, not new code: cross ↓ is a local-judge driver pointed at a remote worker's claims; cross ↑ is the trust ladder with a remote judge registered above the local one and the independence floor routing the residue upward.

## 6. Steelmen

### 6.1 "Independence is unmeasurable — two different models can still share a training corpus / a failure mode; a digest comparison is security theater."
Correct, and the note does not claim otherwise. A different weights digest is *necessary, not sufficient* for independence — two models trained on overlapping data share blind spots a digest cannot see (the same way 115 says pinned weights ≠ pinned behavior). The defense is the *floor direction*: the independence coordinate can only ever make a verdict *more* conservative (downgrade AGREE → ABSTAIN), never license an AGREE on its own. So a *false* "INDEPENDENT" reading (two models that look distinct but share a blind spot) is no worse than today's behavior (where the seam assumes independence unconditionally), while a *true* "SAME_GENERATOR" reading correctly blocks the worst case (literal self-grading). The coordinate is a **cheap lower bound on distrust**, not a proof of independence — which is exactly the posture of every other DOS floor (the prefix-overlap floor over-approximates collision; the grep oracle under-approximates ship). It tightens the worst case and is honest about the ceiling.

### 6.2 "Just always use a remote judge — then independence is free and there is nothing to build."
This is the cloud-only default, and for an *online* operator it is often right (§3.1's cross ↓ is precisely "keep the judge remote"). But it fails the two cases this note's siblings care about: (a) the **offline / on-device** agent ([`121`](121_first-class-on-devices-and-unattended.md)/[`122`](122_the-core-go-runtime-and-the-on-device-kernel.md)) has *no* remote judge reachable in the offline window — the only judge available is a local one, and the danger cell is forced, so measuring independence (and honestly abstaining when it is absent) is the *only* safe move; and (b) the **private / sovereign** operator who *will not* send the claim or its evidence to a vendor — for them the judge *must* be local, and again the danger cell is forced. "Always remote" is a fine default exactly where it is available; the seam exists for the deployments where it is not. (Symmetry with 121 §8: the deferred-client model is the right steady state online; the local rung is what keeps the offline interval from being a trust vacuum.)

### 6.3 "A local model is a worker, not a judge — DOS adjudicates effects with git, it doesn't need the model to judge anything."
The deterministic oracle is always first and always preferred (`judges.py` discipline 1) — where git/EnvPrint/the file-state cross-check can settle a claim, no model is consulted at all, local or remote. The JUDGE rung is *only* the residue the oracle ABSTAINED on (`UNCLASSIFIED` — a legacy reason, a free-text justification, an external assertion with no mechanical cross-check). That residue is real and non-empty (it is the whole reason `llm_judge` exists), and on a device it is *larger* (no git → the oracle abstains more, §121). So the question "which model judges the residue, and is it independent of the worker that produced it" is not optional speculation — it is the live question the moment any JUDGE-rung occupant runs, and a local model is the occupant a private or offline deployment is forced to use. This note governs *that* occupant; it does not propose letting a model judge what git can.

## 7. Litmus tests (acceptance gates)

Each pins a property the same way the existing seam tests do (`test_judges.py` / `test_overlap_policy.py` / `test_env_print.py`).

- **`test_same_generator_judge_cannot_agree`** — a judge whose producer `EnvPrint.digest` equals the worker's (same model grading its own claim) is structurally downgraded from AGREE to ABSTAIN with a typed "no independent witness" reason; it can never auto-clear. (The fail-to-abstain discipline extended from "judge failed" to "judge not independent" — the dual of `test_overlap_policy`'s lying-admit proof.)
- **`test_independence_floor_only_tightens`** — for every `independence` value, the floor can only downgrade a verdict (AGREE→ABSTAIN), never upgrade one (ABSTAIN→AGREE); a hostile independence scorer cannot manufacture an AGREE. (`admissible_under_floor`, re-aimed.)
- **`test_unknown_independence_is_conservative`** — a workspace that stamps no model identity yields `independence=UNKNOWN`, which cannot AGREE on a claim flagged as needing independence; the seam degrades to today's-safe, never to a silent clear. (The missing-`EnvPrint`-reads-as-not-recorded posture.)
- **`test_local_judge_verdict_carries_reproduction_print`** — a local-model judge's verdict stamps the producer `EnvPrint` (model digest, quant, seed, temp); the verdict is re-runnable from the recorded print. A cloud judge's verdict honestly carries no such print (unversioned). (§4.3 — the EnvPrint stamp on the verdict.)
- **`test_cross_locality_escalation_routes_up`** — given a local judge that returns `independence=SAME_HOST` and a remote judge registered above it, a claim the local judge cannot independently clear escalates to the remote judge, not to AGREE. (§3.2 cross ↑, the trust ladder within JUDGE.)
- **`test_model_transport_is_driver_only`** — the `cli`/`socket`/`inproc` transport choice lives in a `drivers/` judge; no module under `src/dos/` (except `drivers/`) names a model endpoint, a socket, or an FFI symbol. (The existing `no dos.drivers import` litmus, extended to the transports.)
- **`test_doctor_reports_judge_independence`** — `dos doctor` lists, per configured judge, its model identity and the `independence` it would have against the declared worker model, so the honesty floor (which judges can actually clear a claim here) is visible, not hidden. (The 121 operator-surface discipline.)
- **`test_judges_public_shape_additive`** — the independence handles are *optional* additions to `Claim`/`JudgeVerdict`; the existing `test_judges.py` suite stays green unchanged and a judge that reports no producer print behaves exactly as today. (The additive-only exit criterion, the `cost`-field precedent.)

## 8. What this note claims, and what it does not

- **Does claim:** the kernel's only coupling to a model is one shell-out that silently assumes the model is *remote*, and that assumption inverts on a co-resident local model (§2); locality is a *measurable coordinate of independence*, governed by the [`117`](185_native-log-adapters-and-the-actor-witness-split.md) witness-≠-actor law, and the seam mis-models it in both directions — under-warning the danger of a shared generator and under-using the gift of a stampable identity (§2). Cross-locality validation (local-judges-remote / remote-judges-local) is the *deliberate maximization* of that coordinate, and the two crosses are asymmetric — sovereignty vs escalation (§3). The danger cell (same generator) and the contended cell (same accelerator) coincide, so the cross-locality move fixes both at once (§3.3). The whole fix is four existing seams re-aimed — `ModelTransport` (the 122 invocation split), the independence floor (overlap's `admissible_under_floor` + judges' fail-to-abstain), the `EnvPrint` producer-stamp (115, which pre-admitted the model object), and *not* building accelerator arbitration because the floor steers out of the contended cell (§4–§5).
- **Does not claim:** that a digest comparison *proves* independence (§6.1 — it is a cheap lower bound on distrust, necessary-not-sufficient; two distinct models can share a blind spot), that a local judge's verdict is *behaviorally* reproducible (§4.3 inherits 115's pinned-weights-≠-pinned-behavior caveat — it is *re-runnable*, not *deterministic*), that DOS should let a model judge what git can settle (§6.3 — the oracle is always first), or that any of this is built (it composes existing seams behind a floor that can only refuse-more).

The meta-answer to the prompt: **the deeper interconnection between DOS and a local host model is not a faster pipe to the inference engine — it is the kernel learning that *where a model runs is a trust fact*. A co-resident model is the case where "the part that doesn't believe the agents" is most at risk of believing a model's verdict on its own work, and simultaneously the first case where a model's verdict can be pinned and re-run. Cross-locality validation — auditing a remote model with a local one you control, or escalating a local model's residue to a remote one it cannot have touched — is how you make the independence coordinate large on purpose. And the kernel already owns every seam this needs: it has only to stop treating the model as an opaque CLI string and start treating it as a stamped, floored, independence-measured participant in the trust ladder it already runs.**

## 9. Implementation handoff — the turnkey contract for the next node

This section is the buildable contract for a downstream agent (an *ultracode* node). It was hardened by a multi-agent planning pass (4 design passes + 2 adversarial passes); the blockers the skeptics found are pinned **into the phase text below as normative decisions**, not left open. Read §1–§8 for *why*; this section is *what to build, in what order, and the exact traps to not trip.*

### 9.0 Scope — what this slice does and does NOT do (read first; the headline correction)

The adversarial pass found the design's most important over-claim, and it is corrected here: **this slice wires the independence floor into the EVAL instrument (`dos judge-eval` → `judge_eval.compose_deterministic_first` / `score`), NOT the live `dos judge` / decisions path.** Verified: the live path calls `run_judge`/`.rule` directly and has no `compose_deterministic_first` call site (`cli.py`/`decisions.py` contain none). Therefore:

- **This slice makes the independence handle EXIST** (the kernel fields), **makes the EVAL harness HONOR it** (the floor, measurable via `false_clear_rate`), and **ships the reference local driver** (`OllamaJudge`). That is a complete, valuable, *additive* unit.
- **It does NOT make "every local-model config safe end-to-end."** Populating `Claim.producer_print` at the live decisions boundary and flooring the live `run_judge` path is a **named follow-on (Phase L, §9.3)** — out of scope here, called out so the next node does not believe §2–§8's aspirational framing is fully realized by this slice. §8's "Does claim" line should be read as *the design*; §9 is *the buildable subset of it.*

Every change is **additive**: a workspace that stamps nothing behaves exactly as today (the `cost`-field precedent — `JudgeVerdict.cost` is the proof that an optional, safe-default field rides this dataclass without disturbing the suite).

### 9.1 The four blockers, pre-resolved (pin these before writing code)

These are the decisions a fresh node *cannot guess* and that both adversarial passes independently flagged. They are normative:

1. **The floor fires only on a POSITIVE non-independence finding; UNKNOWN is left untouched by default.** A strict "UNKNOWN cannot AGREE" floor would red every existing `_Heur` AGREE in `test_judge_eval.py` (those carry no producer-print → UNKNOWN). So: `apply_independence_floor` downgrades AGREE→ABSTAIN **only** when the finding is `SAME_GENERATOR` or `SAME_PLATFORM`. `UNKNOWN` (either print absent — nothing to compare) is **left as-is**, preserving the green baseline. The strict "UNKNOWN also cannot clear" behavior lives behind an **opt-in** `require_independence: bool = False` parameter — off by default. This reconciles `test_unknown_independence_is_conservative` (which asserts the *strict-mode* behavior) with additivity: the test exercises `require_independence=True`.
2. **The `to_dict` shape change is ONE atomic edit with its canary test.** `tests/test_judges.py:60` asserts `set(v.to_dict()) == {"stance","why","evidence","cost"}` by **equality** (verified). Adding keys reds it — and a naive node would "fix" it by reverting the stamp, silently deleting the whole feature. So Phase 1's step text **mandates**: update line 60 to `== {"stance","why","evidence","cost","producer_print","independence"}` *in the same commit as the `to_dict` change*. It is the additive-shape canary, not a regression.
3. **The `model_transport_is_driver_only` litmus is AST-based and forbids only DRIVER-SPECIFIC literals — never the bare tokens `socket`/`transport`.** Verified red-on-arrival hazard: `lane_lease.py:710` does `import socket; socket.gethostname()` (legitimate kernel code) and `run_id.py:229` has `transport` in a comment. A grep-text or bare-token litmus reds immediately (the exact docs/117 trap). So the test: (a) uses **AST**, the `tests/test_vendor_agnostic_kernel.py` shape (`ast.parse`, collect `Name.id`/`Attribute.attr`/`Import`/`ImportFrom`); (b) forbids only literals that **cannot occur in legitimate kernel code** — `ollama`, `11434` / `:11434`, `llama_cpp` (or the exact FFI symbol), and the driver module name `ollama_judge`; (c) globs `*.py` (NOT `rglob`) so `drivers/` is excluded. It does NOT ban `socket` or `transport`.
4. **`EnvPrint` is imported under `TYPE_CHECKING` in `judges.py` (anti-cycle).** `env_print.py:74` does a top-level `from dos import durable_schema`, and `dos/__init__` imports `config` which imports `env_print` — a top-level `from dos.env_print import EnvPrint` in `judges.py` risks a partial-init cycle. Use a string annotation `producer_print: 'EnvPrint | None' = None` under `if TYPE_CHECKING:`. The field is never *constructed* inside `judges.py` (drivers/eval build the prints), so no runtime import is needed — the same lazy posture `env_print` itself uses for `dos.__version__`.

Plus five MEDIUM corrections folded into the phases below: **(5)** the AGREE→ABSTAIN downgrade must **preserve `evidence`/`cost`/`producer_print`** (`JudgeVerdict.abstain(reason, evidence=v.evidence, cost=v.cost, producer_print=v.producer_print, independence=ind)`; use `dataclasses.replace` on the non-downgrade stamp path); **(6)** rename the middle bucket `SAME_HOST`→`SAME_PLATFORM` (it is `EnvPrint.platform` = OS+arch, a *coarse* signal, not a host identity — only digest-equal `SAME_GENERATOR` is a *strong* independence-violation signal); **(7)** the floor **recomputes** `independence` from the prints and **ignores** any value a judge driver supplied (the docs/107 read-path-re-adjudicate lesson — the stamp is judge-driver-written, hence distrusted; pin `test_floor_recomputes_independence_ignoring_supplied_value`); **(8)** `independence` is a bare enum-string defaulting to `'UNKNOWN'` (the `cost`/`engine` precedent — trivial `to_dict`, no enum import in the leaf), with an `Independence(str, enum.Enum)` for the comparison helper's return; **(9)** implement the digest/platform comparison **inline** in `judge_eval._independence_from_prints` — do NOT import or hook the docs/115 §5 `FLEET_ENV_MISMATCH` gate (it does not exist yet; this is a *sibling* implementation of the same comparison, refactor-to-share later if the gate lands).
