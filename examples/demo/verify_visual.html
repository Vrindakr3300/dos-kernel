DOS— the kernel that doesn’t believe the agents

The money-moment, stepped through

An agent claims two units of work shipped. One did; one didn't. DOS never reads the agent's claim — it reads git history and answers from what actually landed. The whole pitch is the contrast between the two verdicts. every line is real output of examples/demo/verify_demo.sh

What the agent CLAIMS

agent · claim #1
“Shipped AUTH1 — the login endpoint is done.”
agent · claim #2
“AUTH2 is done too — all work completed!”
Both are just sentences the agent typed. Equally confident, equally cheap. Nothing about the claim itself tells you which is true — that's the problem DOS exists to solve.

What git actually RECORDS

e389e8b AUTH1: ship the login endpoint
— — — — (no commit mentions AUTH2)
the git history is the ledger — one real commit, nothing else.
$ dos verify AUTH …
Press Step ▶ to verify each claim against git.

Why this is the whole idea. An agent can claim AUTH2 is done all day long; dos verify just reports what the artifacts say — and they say it isn't. The little via grep-subject / via none tag tells you how it knows: it found the phase token in a commit subject, or it found it nowhere. Two things make this trivial to drop into a pipeline:

  • verify needs no plan, no registry, no config — hand it a plain git repo and it answers from history alone.
  • the exit code IS the verdict (0 shipped / 1 not), so any shell or CI step branches on it without parsing a word — replace the step that trusts an agent's "done" with dos verify PLAN PHASE.

Reproduce every line: bash examples/demo/verify_demo.sh (or verify_demo.ps1).
