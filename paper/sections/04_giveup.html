The one action that helps: give up correctly

§{{sec:boundary}} drew the line: spending the verdict in-loop is flat-to-harmful, and the one in-loop action that paid off was the negative one. This section measures it. A byte-clean detector (§{{sec:bench}}) raises an alarm but must be careful what it does next: most actions cross from detecting into fixing, and a fix authors a correction the agent will trust — breaking the byte-clean rule the detector was built on. Exactly one action does not cross that line: withhold compute. Stop a clearly-doomed run and hand its remaining budget to one that might still win. Stopping authors no correction, so it cannot mislead. Of every action we measured, this negative one was the only survivor.

Error-gated give-up is accuracy-free

The gate is simple. It watches each tool the agent uses. When a single tool returns K structured errors in a row — errors written by the environment, not by the agent — and its most recent call is still erroring, the gate proposes a halt. The second condition matters: the run must be failing now, not have failed earlier and recovered.

The question is whether this ever stops a run that would have won — a false-abandon, the gate halting a run that would actually have succeeded. It is the only costly mistake the gate can make, so it is the number that has to be zero.

At K≥3 the error-gated gate false-abandons zero winners — across all 22 models, frontier to weak. Its Wilson-95 upper bound is just 0.0023, so even the worst the true rate could plausibly be is near zero. The same threshold is sound on a second, independent benchmark, EnterpriseOps-Gym (a ServiceNow operations gym run on gemini-2.5-flash): K≥3 false-abandons zero winners on both arms there, while K=2 is unsound (it halts one winner). Two different benchmarks, one safe threshold.

The naive gate — halt on a bare repeat, no error condition — false-abandons 6 winners at K=3, every one the same model (grok-4) doing the same harmless thing: polling “is it ready yet?” until the resource became ready and the run succeeded. A repeated answer is not an error. The byte-clean rule saves the gate: the signal is the environment-authored error, not the repeat — “repeated tool call → halt” kills pollers, “repeated tool error → halt” does not, because a poller's repeats are not errors and the agent cannot forge the error bytes.

So blunt a gate works because, on the K statistic — how many errors a tool stacks up in a row — the winning runs and the doomed runs fall into almost non-overlapping ranges: winners rarely stack errors, doomed runs do. That a clean class separation collapses to a threshold test is nothing new; it is the separable regime of any threshold classifier, and the SPRT/CUSUM tradition (below) is built on exactly such run-length count statistics. The part that matters here is the statistic's input: the count is over environment-authored error bytes the agent cannot forge. That is what makes the integer threshold at K≥3 both sound (zero false-abandon, measured) and non-gameable. You do not need a probabilistic model when the two populations barely touch and the statistic separating them is one the agent cannot author.

The saving follows. At K=3 the gate fires on ~7% of runs (the thrashing tail) and on each suppresses a median ~53% of its tool calls (the doomed back half) — pooled across both EnterpriseOps-Gym arms, 11.4% of total tokens, saved at zero false-abandon. Across five recorded run-sets it saved 3.5k–43.7k real tokens, false-abandon 0.000 every time.

No novelty in the mechanism; the contribution is the grounding

The mechanism is old and we claim none of it: our gate is the trip half of a circuit breaker (Nygard 2007), the degenerate corner of an SPRT (Wald 1945{{cite:wald}}), and a CUSUM (Page 1954{{cite:page}}) collapsed to a plain error tally. What is new is the grounding: (i) it counts only environment-authored bytes the agent cannot forge; (ii) the threshold is calibrated against a measured false-abandon cost, not a guess; and (iii) it stays advisory — it proposes a halt, never trips one — so a mis-set K degrades to a few wasted tokens, never a killed winner.

Every active fix failed

Give-up is what was left standing after we tried the active fixes too — the ones that author a correction — on the cross-benchmark target (EnterpriseOps-Gym, gemini-2.5-flash; these numbers stay out of the Toolathlon tables and imply no Toolathlon lift).

  • WARN the agent (re-surface the alarm into the loop): flat on the natural failure stream, +0.20pp. The +6.2pp lift this verb shows elsewhere held only when failures were artificially injected, not in the wild.
  • Rewind (content-free surgery — cut the agent's bad turns back out of the transcript): 0 conversions, 0 flips on 20 paired fired tasks. The null only got stronger as we added data (44 → 103 → 240 runs per arm).
  • The byte-clean cure (re-surface the environment's own schema correction — the cleanest fix we could build): wired live, it fired on 77 of 88 runs, and it was net-negative: −5 task successes overall, −7 on the no-thrash slice (sign-test p=0.016; successes fell 7 → 2 of 88). It converted the 2 runs it was aimed at — but injecting a corrective turn at all perturbed runs that would have passed on their own.
The harm is the intervention's existence in the loop, not the content of its bytes. Even a perfectly byte-clean correction perturbs runs that were going to win. Only the negative action — withhold compute — authors no turn. It perturbs nothing, and it is the only fix that survives a measured bake-off.

Honest ceiling

This result proves specificity, not sensitivity: wherever the gate fires it never halts a winner, but how often it fires we cannot promise. On Toolathlon at the sound threshold it fires on not a single run (one fire at K=2), because a capable model that fails silently emits no error to count — the fires, and the saving, come from EnterpriseOps-Gym (31 fires at K=3, none on a winner). So the claim is narrow: the gate is accuracy-free wherever it fires, but the 11.4% token saving is a single-corpus, single-model figure. The specificity is the durable, regime-independent result; the saving is real but travels less.
