Appendix C. How to reproduce — every number, command, and released item

This appendix lists every result the body claims, the command that regenerates it, the artifact that backs it, and what you need to run it — so a stranger with the repository and one API key can re-derive the numbers, or see exactly what stands between them and a gated one. The frozen results are $0 and deterministic (a shipped CSV or corpus); the live results are paid (an external benchmark through a model API). The distinction is the methodological spine: a frozen replay gives a rate, only a live run gives an intervention payoff.

C.1 The reproduction principle: re-derive, never trust the cache

One rule governs every script below, aimed at our own pipeline: every number is recomputed from the byte the agent could not author, never read back from a stored summary. The frozen results recompute from a per-run CSV the benchmark's own grader wrote; the live results recompute each verdict from the agent's stored message with the current extractor, not the cached bit. This is not pedantry — in the cross-model run (§{{sec:payoff.secondmodel}}) a naive sum of cached bits reported the stronger model at J = 4, and re-deriving from the stored message gave the true J = 5, recovering an over-claim the older extractor had missed. So each command ends in a fold that re-reads ground truth, and several ship a --check mode that exits non-zero on drift.

C.2 One-time setup

Everything runs from a single editable install of the kernel plus its benchmark extras. The kernel's own dependency set is PyYAML-only; the paid runs pull in the external benchmark and a model client.

Two setup traps cost real time. tau2-bench transitively imports audioop (removed in Python 3.13) — install the audioop-lts backport or the import fails before any task runs. And a Gemini access-token key (AQ.-prefixed) expires; re-smoke it before any batch (models?pageSize=1&key=…), or a dead key silently zeroes the run and makes “we tried, it found nothing” look like a result.

C.3 The two $0 results (frozen, deterministic, no key)

These read a shipped CSV and recompute the paper's numbers with no network, no model, no spend. They are the detector-quality story (§{{sec:result}} / Appendix A.3) and the give-up story (§{{sec:giveup}} / Appendix A.3). Each is its own single source of truth and ships a --check mode that fails loud on drift.

The durable-rows CSV carries a fingerprint, and the paper records it: the rows this paper's numbers were drawn from fingerprint to 1a55e8d8e2d4 (additivity.rows_fingerprint()). The paper's own build step warns if the live CSV has drifted from that value, so a stale paper cannot quietly ship wrong provenance. If you regenerate the rows and the fingerprint moves, the numbers move with it — that is the point.

C.4 The live results (paid, gated behind a key)

These are the headline of the paper — the out-of-loop payoff, run live (§{{sec:payoff}}). They drive a real agent through tau2-bench, read the environment's own database hash, and count an integer J of corruptions a referee actually prevented. Both the agent and the user-simulator are live models, so cost scales with dialogue length; the whole program in the paper ran for well under the budgets below. Every live-output directory is gitignored on purpose — the seed configs embed the API key — so the repository ships the drivers that regenerate the result, not the raw transcripts. The $0 synthetic smoke that each driver ships proves the mechanism with hand-built tool-calls and no model; run it first, every time.

Two model-specific details silently zero a run if missed. First, the cross-model run needs a per-tier reasoning setting: the flag that suppresses a flash-only crash on long dialogues is fatal to the thinking-only pro tier (“Budget 0 is invalid”) — a pro attempt that re-used the flash flag errored on all 60 tasks and reported a false J = 0 (a regression test test_model_args.py now pins it). Second, the coordination run must give each task the reservation's real owner as the user identity, or a correct agent rightly refuses to touch a stranger's booking and the designed conflict never fires. Both are harness details that make “we tried, it found nothing” an artifact rather than a result.

The complete, append-only run journals for the live program — every checkpoint, every trap, every dollar — live next to the drivers and are the most faithful record of how the numbers were produced: benchmark/agentprocessbench/writeadmit/RUN_PLAN_live.md (the write-admission run, J = 5 → the design notes that became docs/228), and the matching design notes docs/232 (cross-model, J = 10) and docs/233 (coordination, J = 6).

The Appendix D programs reproduce the same way, from their own drivers and journals (Table C4). Each follows the Table C3 pattern: a keyless command that proves the mechanism, then a gated command that reproduces the live number.

C.5 Released items

The full inventory — the kernel (src/dos/; distribution dos-kernel, import/CLI dos), the seven-benchmark suite (python -m benchmark._run list prints each one's prerequisites and cost; the three the paper draws on are toolathlon, agentprocessbench, and enterpriseops), the design notes, and the paper's own reproducible build (python paper/build.py) — ships in the project repository. The dependency arrow is one-way: the benchmarks and the paper import dos / call the dos CLI; nothing under src/dos/ depends on them. The write-admission floor both live consumers stand on is src/dos/effect_witness.py (the believe_under_floor discipline of Appendix B.2); the build is single-sourced from paper/meta.py, with every figure regenerated from the run data.

The shortest path from a clean clone to the numbers: three commands recompute the frozen rates from shipped data at $0 (Table C2); the live programs (Tables C3–C4) drive real agents through tau2-bench and Agent-Diff off a single key to recompute the intervention payoffs — and every one re-derives its number from the byte the agent could not author, never from a cached summary.
