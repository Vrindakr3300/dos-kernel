Is the signal real? Naive baselines

A detector that fires on the right runs is only impressive if a dumb detector cannot do the same. So we measured the dumb ones, scored by the same lift and false-alarm from §{{sec:bench}} (lift = precision minus the 76.2% base fail rate; false-alarm = how often the detector fires on a run that actually passed). A good detector has high lift and low false-alarm.

The recall mirage. Recall is the wrong scoreboard here. The baseline that prints "fail" on every run posts 100% recall — it catches every failure because it flags everything — yet its lift is +0.0pp and it false-alarms 100% of the time. Score by lift and false-alarm instead and it falls to the bottom where it belongs. Our detectors make the opposite trade (high precision, low false-alarm, low recall) because a verdict is meant to be acted on, and §{{sec:giveup}} is what that looks like: you only give up on a run you are sure is doomed.

The tight grammar is load-bearing. You might think reading the structured error envelope more loosely would catch more failures. It does not: loosening buys ~5pp of precision but pushes false-alarm from 0.24% to 25.6%–69.4% (≈105×–285× worse). You cannot buy recall by reading the bytes you already have more loosely; the only way to catch more is an independent witness — exactly the Tier-B re-read of §{{sec:payoff}}. The per-model grid (Fig. {{fig:permodel}}) shows the precision holding across every model that fires.

Positioning

Three kinds of system sit near this work. The first does the same job by a cheaper-looking trick — re-run the agent until a check passes — and works only for one agent. The second does the same job — catch a silent failure — by a different method (a trained classifier). The third does a different job entirely. We take each on its own terms.

The re-run wrapper (and why a fleet defeats it)

The most common reaction is that this looks unnecessary: if you do not trust an agent’s “done,” just check the result and re-run when the check fails. The loop is correct for a single agent and we make no claim against it — only against the dismissal it smuggles in, that single-agent reliability is therefore trivial. Two parts of the loop are real work the phrase hides: the check must be one the agent cannot forge (a git-ancestry, registry-first ship oracle under a closed vocabulary, not “re-read the result,” which falls into the same mirror-verifier trap), and the loop must sense fast enough to gate every turn (here the in-flight path needed a rewrite from a ~250 ms Python hook to a ~10 ms native one). Neither is a few lines of glue.

But the headline claim — the place a re-run wrapper provably cannot follow — is the concurrent one, and we concede the stronger objection first. If the shared store offered a transactional primitive (a serializable transaction, a compare-and-swap, a conditional write), a wrapper around even a lone agent could fuse check and write into one atomic step and close the race with no referee. Our claim is scoped to the regime that holds for agents in the wild: they act through high-level tool APIs (cancel a reservation, file a ticket) that expose only unconditional writes, so no such primitive is available to wrap. There the TOCTOU argument bites — A’s read and write are separate non-atomic calls, so no number of re-runs makes two calls one. The number settles it: §{{sec:payoff.race}}’s arbiter prevents all 6 live would-be clobbers across its 8 racing pairs and a re-run wrapper prevents none, because each agent’s own check passed. (The arbiter is a lock service supplying the serialization the store does not, lifted outside any single agent — which is why it works where a wrapper on A cannot.)

The honest converse is our own boundary. At the single easy hop where a wrapper does apply, handing the verdict back in-loop is mildly harmful (ΔB, Appendix D) — so we do not rest the headline payoff on the single-agent case. The concurrency case without a transactional store is where the entire incremental payoff over a wrapper lives.

Trained classifiers

The nearest neighbour is a trained classifier: a model fitted on a labelled set of past failures, which then predicts whether a new run failed (arXiv 2511.04032{{cite:silentfailures}}, 2025, about 98% accuracy on its own data). On paper that beats our detectors by a wide margin. The catch is what the number costs, and where that 98% actually lives.

A classifier needs a labelled set of failures from the same kind of task, and a held-out slice to score it honestly. That labelled set is exactly the thing you do not have on a task you have never seen, which is the whole problem. Run the classifier fairly on this corpus and two things happen. Its best-balanced operating point just predicts "everything failed": on a benchmark where 76.2% of runs fail, that scores well by accuracy but buys no real skill, and this is where the 98% headline lives (the recall mirage from §{{sec:baselines}}). Its only deployable operating point — the one that almost never cries wolf — fires exactly once. So at a false-alarm rate you could ship, it does nothing.

The two classifier rows are the ends of one dial. At the deployable point (the only setting that does not constantly cry wolf) it fires once in 6,862 runs — effectively silent. At the F1-optimal point (best by the usual accuracy measures) it just predicts “everything failed”: 100% fire, +0.0 lift, no skill. The widely-quoted “98% accuracy” lives at that second setting — the recall mirage of §{{sec:baselines}} in classifier's clothes. No dial position both fires usefully and stays trustworthy on an unseen task; our zero-training detectors sit at +16 to +19 lift at a deployable false-alarm rate.

Two further problems. The classifier reads the agent's own trajectory structure, so it risks the mirror-verifier trap (§{{sec:intro}}) — it can learn the tells a model stops emitting once trained against. And when we feed it the DOS detector flags as features, its top weights are those detectors (dangling_fired, terminal_error_fired, tool_stream_fired): it re-derives the cheap signals it was handed. The right place for such a model is not beside DOS as a rival but under it — a JUDGE-rung adjudicator (ORACLE → JUDGE → HUMAN) that rules only on the residue the deterministic detectors abstained on, fail-to-abstain, never the floor.

Arbiter neighbours

Two multi-agent arbitration systems sit nearby: Limen{{cite:limen}} (advisory write-leases) and CodeCRDT{{cite:codecrdt}} (lock-free CRDT). Both compare to DOS's arbitrate() call, not its detector, and neither can run on this benchmark: a frozen single-agent trace has no live write to mediate. (The arbiter axis is not unmeasured in this paper — §{{sec:payoff.race}} exercises it live and counts J = 6; neither neighbour reports a comparable prevented-corruption count.) On this frozen target DOS's arbiter likewise has nothing to mediate, and its enforcement point — a pre-tool permissionDecision: deny hook an agent runtime such as Claude Code honours before a tool runs — is never exercised by a single-agent benchmark. The detector axis of this study and the arbiter axis of these neighbours do not share a scoreboard.

Limitations

  • In-loop: DETECT, not FIX — no task-lift number on Toolathlon. Every Toolathlon figure is detection on frozen runs; the one validated in-loop action is give-up (§{{sec:giveup}}), which repairs nothing. The out-of-loop result (§{{sec:payoff}}, J = 10 across two models / 120 tasks) is a measured existence result at an 8.3% rate, not a calibrated population rate, and on a different benchmark (tau2-bench).
  • The recall ceiling is hard and tightens on the frontier. Union recall is 6.18%–7.42%, so ~93% of failures stay silent, and fire-rate falls toward zero on the strongest models.
  • Four syscalls are null by construction here. Toolathlon runs one agent on isolated state with no git or WAL, so arbitrate, verify, liveness, and resume have no job; only the in-flight detectors do.
  • The give-up token-saving magnitude is single-corpus, single-model. The gate never wrongly halts a winner where it fires (specificity, across 22 models and two benchmarks); how often it fires, and so how much it saves, is benchmark-dependent.

Conclusion

Hold one thing fixed — a byte-clean verdict the agent cannot forge, scored by a third-party oracle DOS does not own — and the result is about where to spend it. In-loop, every active fix (warn, rewind, inject the environment’s own correction) was flat or net-negative, because injecting a turn perturbs runs that would have passed; the one in-loop action that helped was the negative one, give up correctly (zero of 1,634 winners halted across 22 models, ~11% of tokens saved). Out-of-loop, the same verdict pays on both ways a fleet fails — a write-admission gate blocked 10 over-claims at an identical 8.3% rate across two models (so the signal does not fade at the stronger tier), and an arbiter prevented all 6 would-be clobbers across 8 racing pairs — both off a byte no agent wrote, neither touching the judged agent. The decisive move is not a better detector but a better consumer: detect cheaply in-loop, stop the doomed run, and spend the verdict out-of-loop — with the W3 residue (a goal with no world footprint) still beyond any witness short of a human.

Directions

The result generalizes along the axis that gave it: where a non-forgeable verdict is spent, and what witness it can read. Four directions follow from the honest edges above, each a measurement we know how to run. Close the RLVR loop to a weight update. §{{sec:payoff.scope}} measures the label purge (poison reward labels removed off ground truth); the next rung is the trained-behaviour delta — fine-tune two models on the poison and the witness-cleaned reward set and measure whether the second over-claims less. The pipeline is built and the proxy separates (+60pp over a control that over-claims 0/4); only the weight update is unmeasured, and that is the one number a training lab would want first. Raise the witness floor. About 38% of frontier goals reach no sound witness at any rung (§{{sec:boundary}}); the open question is how far a richer, still-non-forgeable witness (a structured state-diff, an environment-authored trace, a second independent grader) can push that floor down before the irreducible human-seed residue. Cascade depth as a calibrated curve, not an existence shape. Appendix D shows the payoff compounds FD across a branching fleet, but with an injected root and per-node re-inheritance; a fully serial chain whose nodes write on the poison turns the shape into a calibrated constant. Push the referee from advisory to enforcing, carefully. Everything here is advisory — it proposes, a human or a downstream system decides — which is the right default; the controlled question is where an out-of-loop enforcement point (a pre-write deny the runtime honours) can be trusted to act without a human in the loop, and where it cannot. The throughline is a small trusted referee that lets a fleet of fallible, self-narrating agents — and the loop that trains them — be scaled without scaling the trust placed in any one of them.

Reproducibility

Every number in the Toolathlon study (the detectors and their ceiling, §{{sec:bench}}–{{sec:result}}; the give-up specificity, §{{sec:giveup}}; and the naive baselines, §{{sec:baselines}}) recomputes offline, with no network or model calls, from the durable per-run join replay_all_rows.csv (7,116 records, content fingerprint sha256:1a55e8d8e2d4 — computed line-ending-normalized, CRLF→LF, so a checkout on Windows whose raw sha256sum differs is not seeing drift; additivity.py:rows_fingerprint is the canonical computation). The additivity single-source-of-truth re-asserts the headline claims and exits non-zero if a regenerated file drifts from them, so a claim cannot silently part from its data; the give-up cross-benchmark numbers and figure regenerate from their own script. The one exception is §{{sec:payoff}}: it reports a live, paid run (tau2-bench through Gemini; single-model run {{fact:spend_writeadmit_single}}, cross-model run {{fact:spend_writeadmit}}), so its numbers are a fixed read-off of that run, not an offline recompute. They are transcribed verbatim into the figure's single source of truth (paper/_VERIFIED_FACTS_228_2026-06-08.md and _VERIFIED_FACTS_232_2026-06-08.md), and the run is resumable from a per-task cache so it never re-spends; re-running it requires a Gemini API key and reproduces an existence result (the gate blocks real over-claims), not the exact count, since the live rollouts vary.
