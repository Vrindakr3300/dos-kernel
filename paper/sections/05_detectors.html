The benchmark and the three detectors

We test on Toolathlon, a hard benchmark for agents that use real tools across many apps over a long task. The property that makes it useful to us is its grader. Toolathlon ships its own oracle: a separate grading program (evaluation/main.py) that reads the live state of the apps the agent touched and exits 0 for pass or 1 for fail. The oracle is not the agent and does not read the agent's words; by construction it judges results, not narration. The benchmark is hard: across the corpus a run fails 76.2% of the time. We call 76.2% the base fail rate.

We do not run new agents. We replay frozen traces. The public Toolathlon-Trajectories corpus holds 7,116 recorded runs (22 models × 3 runs over ~108 tasks); 6,862 carry an oracle label (5,228 fail, 1,634 pass). The remaining 254 have no oracle verdict, so we drop them — we never guess a label. A detector is a pure function of a recorded run, so we can replay the whole corpus offline, with no live agents and no model calls. Every number below recomputes from the recordings alone.

The three byte-clean detectors

Each detector obeys the byte-clean rule from §{{sec:intro}}: it reads only bytes the agent under judgement did not write — the environment's tool results, never the agent's own narration. The three differ only in which environment-authored bytes they watch.

  • dangling_intent — the agent's last message admits open work ("I still need to…"), and nothing environment-authored ran after it. The agent said it was not done, then stopped.
  • tool_stream — the same call returns the same result N times in a row: the same tool, the same arguments, the same result fingerprint. The environment keeps answering identically and the agent keeps asking. (A small normalizer strips volatile bytes — timestamps, ids — before fingerprinting; we defer the details to §{{sec:baselines}}.)
  • terminal_error — the run ended on an error. Within the last 3 tool results, the environment returned a structured error envelope — a result the environment itself marks as failed — and the agent never recovered. The agent saying "Email sent successfully!" does not suppress the fire when the environment's last result was an SMTP rejection: the agent does not author that byte, so it cannot talk it away.

We score each detector with five numbers, used throughout §{{sec:result}}:

  • Fire-rate — how often the detector raises a flag, as a fraction of all runs.
  • Precision — when it does fire, how often the run really had failed. High precision means “believe it when it speaks.”
  • Recall — of all the failing runs, how many it managed to catch. Low recall means “it stays silent for most failures.”
  • False-alarm — how often it fires on a run that actually passed. This is the costly mistake: cry wolf and an operator stops listening.
  • Lift — precision is misleading on its own, because this benchmark is hard: 76.2% of all runs fail, so a “detector” that flags every run is “right” 76.2% of the time by default. Lift is how far above that floor a detector’s precision sits (lift = precision − 76.2%) — the part of being right that took skill, not luck. A lift of +20 points means a fire is 20 points more likely to be a real failure than a random run; a lift of 0 is no better than flagging everything. We read lift, not raw precision, throughout.

The detection result, and its ceiling

Table {{tbl:detectors}} reports all five numbers for each detector, plus two combinations: the pair (the first two detectors together) and the trio (all three). Read each row left to right as a sentence: this detector fires on fire% of runs; when it does, it is right precision% of the time, which is lift points of real skill above the 76.2% floor; it catches recall% of all failures; and it wrongly flags a passing run only false-alarm% of the time. The last column (tp/fp) is the raw count behind those rates — true catches over false alarms. The shape below is high-precision and low-recall: a detector you can trust, that rarely speaks.

The trade is the same for all three: when they fire they are right 88–98% of the time, at near-zero false-alarm (0.12–1.22% each, 1.59% for the trio). Each clears the 76.2% base by a real margin — +12.0 to +21.9pp of lift. But recall is small: about 2–3% per detector, 6.18% for the trio together (7.42% at 92.4% precision with one opt-in strictness flag that stops crediting a generic executor’s “recovery” — a later same-tool success that is a different operation, not a fix). The detectors catch the honest-narrating slice and the visibly-looping slice; they miss the run that fails quietly.

terminal_error's catches are almost all new: of its 76 catches, 75 are runs the other two never flagged (99% additive), lifting union recall 4.74%→6.18% (+30% relative) at 92.6% precision. It reads a different environment-authored byte and catches a different failure.

This holds on the strongest models too: on the eight highest-scoring models (passing ≥0.30 of tasks), the first two detectors catch 6 failures between them and terminal_error adds 9 more they missed.

The recall ceiling is real and tightens on the frontier

The ceiling is blunt: the detectors are trustworthy when they fire, and they fire rarely — rarest on exactly the models that matter most. One thing does carry across capability: precision. On every model that fires at all, a fire is almost always a real failure — the quality of the signal generalises from weak models to strong. What collapses is fire-rate, and therefore recall. Strong models fail silently: among the top-4 models' failures, the median run takes 19 tool steps (only 6% stop short), 92% end with substantial final narration (median ~1,004 characters), and 59% confidently claim they finished — "All customers have now received their surveys!" on a run the oracle scores as failed. A confidently-wrong run narrates no open work, loops on nothing, and emits no error envelope, so a byte-clean detector has no environment-authored byte to read. What changes on the frontier is not the detector's quality but its opportunity: the visible failures it can see thin out toward zero.

The two panels of Figure {{fig:capability}} answer two different questions. The left panel is “how much can these detectors help?” — the answer shrinks toward zero on the frontier. The right panel is “when they do help, can you trust them?” — the answer stays yes everywhere. A signal can be highly trustworthy and still rarely available.

So the result is narrow and solid: a byte-clean, zero-training detector that is right when it fires (against a grader we cannot influence) but fires on only a small, shrinking slice of failures. That high-precision, low-recall, shrinks-on-the-frontier shape is the in-loop ceiling, and why the verdict had to be spent out-of-loop (§{{sec:payoff}}): a signal this trustworthy and this narrow earns precisely one in-loop action — give up correctly (§{{sec:giveup}}) — and pays elsewhere only when handed to the rest of the fleet.
