Appendix B. How it works — from a forgeable claim to a grounded refusal

Appendix A said what DOS believes and where a verdict pays. This appendix says how a verdict is computed — the machine that turns a forgeable sentence into a grounded "no" — and shows the two live consumers from §{{sec:payoff}} are the same pipeline pointed at the two ways a fleet fails. The reference points are real: the gate is benchmark/agentprocessbench/writeadmit/gate.py, the referee-between-agents is coord_loop.py, and the floor both stand on is dos.effect_witness.

B.1 The pipeline: four steps, and only one of them decides

Every DOS verdict — the write-admission gate, the race arbiter, every detector here — runs the same four-step pipeline. What does the work is not any one clever step but the strict separation between them.

  1. Gather evidence — at the boundary, never inside the decision. All reading from the messy world happens here and only here (the database hash, the tool-result stream, the git log); it judges nothing, just collects bytes and notes who wrote each one.
  2. Tag each byte with its author. Every piece of evidence is labelled by which rung of trust it came from — agent, environment, or third party. This label decides everything downstream (B.2).
  3. Classify — a pure function, no I/O. One function takes the tagged evidence and a policy and returns a verdict, reading no files and asking no model, so the same evidence returns the same verdict forever. The only step that decides.
  4. Return a typed refusal — from a closed list. One value from a fixed vocabulary (admit / block / abstain, with a named reason), so a consumer branches without parsing prose and a human can check every answer was accounted for.

Splitting reading from deciding is the reason a courtroom separates gathering testimony from rendering a verdict: a decision that re-reads the world while deciding can be swayed by what it re-reads. By the time the pure classifier runs, the evidence is fixed and labelled and the agent has no further move. I/O at the boundary, a pure verdict at the core is the one structural rule the kernel obeys.

B.2 The accountability ladder — why the agent's own words can never tip the scale

Step 2 is where byte-clean turns from a slogan into a working mechanism. Each piece of evidence carries a label naming its author, and the labels are ordered by how hard they are to forge:

  • AGENT‑AUTHORED — the agent wrote it (its final message, its self-report). This is the Vaseline: admitted into evidence but marked, and that mark keeps it from deciding anything.
  • OS‑RECORDED — the environment wrote it (the database hash, the mail server's error envelope, the git commit). The agent triggered the action but did not author the byte.
  • THIRD‑PARTY — an independent grader wrote it (Toolathlon's evaluation/main.py, tau2's database-hash check). Authored by neither the agent nor DOS.

The rule that makes the floor unforgeable: only an OS‑RECORDED or THIRD‑PARTY witness may set the "refuted" bit. An AGENT‑AUTHORED byte can be recorded (the verdict notes the agent claimed success) but can never confirm or refute. So a policy that writes itself a glowing receipt moves nothing — its bytes land on the forgeable floor, treated as no evidence either way; only a byte from a higher rung flips a write from "admit" to "block". In code this is the believe_under_floor discipline: a read-back on the AGENT‑AUTHORED rung yields unwitnessed, never refuted.

The agent's words are admitted as testimony, never as proof: only a byte the agent did not author can decide the case. This is the A.2 trust ladder (ORACLE → JUDGE → HUMAN) seen from the evidence side: an accountable witness ranks above the agent's narration, and a missing witness fails safe to "admit" — trust the unforgeable thing first, and when it is silent, do the cautious thing, never the flattering one.

B.3 The two consumers as one pipeline, two witness-moments

The write-admission gate of §{{sec:payoff}} is the four steps turning. Gather A's final message (the claim) and the tau2 evaluator's db_match result. Tag the message AGENT‑AUTHORED and db_match OS‑RECORDED (True an attestation, False a refutation, None no witness). Classify: does A confidently claim a write (a closed write-verb ∩ mutating-tool test, not a keyword sniff) and does an accountable witness refute it? Block only when both. Return admit / block / abstain. The structure buys three properties: it cannot be talked into admitting (only an OS‑RECORDED byte sets "refuted," so a fake receipt changes nothing — the gate can only block more under a real witness, never less); it never blocks correct work (a database-confirmed write is attested, so all nine honest writes were admitted and only the five refuted ones blocked); and blocking B's input writes nothing into A's loop, so the in-loop harm of §{{sec:giveup.activefix}} is structurally absent — the entire reason the consumer is B, not A.

The race arbiter of §{{sec:payoff.race}} is the same pipeline with one substitution: the witness is read before the write, not after. A reservation is a region reservations/<id>; the pure call arbitrate(region, live_leases) refuses A2's lease on a region A1 holds (the same disjointness verdict the live dos lease-lane uses, no model, no I/O), so A2 — re-run after A1 commits — sees R already cancelled and does nothing. Its scoring witness is the same airline database hash, and a clobber is counted only when directional (the discipline that took the headline from 7 to an honest 6, §{{sec:payoff.race}}). The arbiter's refusal is a refusal in the identical sense the gate's block is: a typed "no" from a pure verdict over evidence the agent cannot forge. One reads the witness after the claim, the other before the write. Two moments, one machine.

B.4 Why the answer is a closed list, not a sentence

The fourth step — return a value from a fixed vocabulary — carries real weight. A free-text verdict ("looks mostly fine, but maybe double-check the booking") cannot be acted on by a machine or audited by a human: you cannot enumerate the prose a model might emit, so you cannot prove you handled every case. A verdict that is one of {admit, block, abstain} with a named reason can be branched on without parsing, logged without ambiguity, and — the part that matters for trust — checked for completeness. This is why "abstain" is a first-class member, not a shrug: a verdict allowed to say "I have no witness for this" never has to guess, and a kernel that never guesses is one whose every "block" you can believe. That closure also lets the same verdict be spent in two places without re-deciding — the gate's "block" and the arbiter's "refuse" are the same typed object a consumer routes (drop the write, hold the lease, surface it) but never re-litigates. That is the mechanism behind the paper's one sentence: "handed to anyone else" is nothing more than routing a closed, grounded value to a consumer who is not the one being judged.

The whole machine: read bytes at the boundary, tag each by its author, decide once with a pure function that only an unforgeable byte can sway, emit one value from a closed list — then route that value anywhere except back to the agent it judged.
