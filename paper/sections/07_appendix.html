Appendix A. DOS in plain words — the Vaseline test

The body defines its terms as it goes. This appendix is for a reader who wants the one idea underneath all of them — do not trust the narrator — in the plainest language, plus two reference tables: a key to every symbol (Table A1) and a plain-language glossary (Table A2). It is self-contained, built around one running picture: a smear of Vaseline on a mirror.

A.1 The one idea, and the Vaseline picture

An AI agent doing a long task also authors the story about it — the actions and the closing summary, and the summary is the problem. It can end with a confident "All customers have now received their surveys!" while the inbox shows zero emails sent: nothing crashed, nothing in the transcript contradicts the happy ending. This is a silent failure, a wrong result in the costume of a right one (on the hard benchmark here three runs in four fail, and nearly every failure is this quiet kind). So the obvious move — read what the agent says about itself — is the mirror-verifier trap: you measure the story, not the world, and training against it just teaches the agent to stop saying the give-away phrase.

The picture: the real state of the world — the inbox, the database row, the file — is a mirror, and the agent's narration is a smear of Vaseline wiped across it. Vaseline changes only what you see, and the smear re-applies for free (the agent can always produce a smoother sentence) — that is what forgeable means. The fix is to read only what the narration cannot coat: the tool results the environment sent back (an error envelope, the identical "pending" returned three times, the absent row) and a third-party judge that re-reads the live world. The agent authored none of those bytes. A check that reads only bytes the judged agent did not author is byte-clean; the test is one question — could the agent have written these bytes? If yes, Vaseline; if no, ground truth.

A.2 What DOS actually is — a referee made of "no"

DOS — Dispatch Operating System — sits between many AI agents and the one shared world they are all changing, the way a computer's OS sits between many programs and one machine, and plays referee. Its motto is blunt: the kernel is the part that does not believe the agents. A referee mostly says one word, no, and DOS is a handful of careful ways to say it — each the same move, a refusal backed by evidence the agent could not forge, pointed at a different question (the map below).

Two features keep the referee honest. The oracle goes first and cannot be argued with — "did this phase ship?" is answered from git history, a record the agent cannot rewrite to flatter itself. The model-judge is on a short leash — used only on what the oracle could not settle, only as advice (it never blocks alone), and when unsure it must say "abstain," never "looks fine," because a model checking a model is still two things on the smeared side of the glass.

A.3 The three detectors, lift vs. recall, and the punchline

This paper measures three of those refusals as detectors (defined in Table A2), each firing on a byte-clean fingerprint of failure: dangling_intent, tool_stream, terminal_error. They produce one triumphant number and one dismal one, both true: when they fire they are right 88–98% of the time (a lift of up to +22 points above the 76% base fail rate), but they catch only ~2% of failures each. Picture a smoke alarm that is 98% accurate but goes off in two of every hundred real fires: run when it sounds, but keep your other alarms. And it sharpens on the frontier — the strongest models fail quietly (a polished, wrong summary, no trace), so the fire-rate drifts toward zero exactly where the leaderboard lives. Quality carries; coverage shrinks.

The punchline is detect, don't fix. In a separate live program every active fix came up empty or worse — warn (flat, +0.2 points), rewind (zero conversions), cure with the environment's own correction (net-negative). The harm is the intervention existing in the loop at all, not its content. The one survivor is the negative action, give up correctly: it fires on a repeated environment-authored error and, across two benchmarks and 1,600+ winners, halted zero winners while saving ~11% of compute. (A naive version that halts on any repeat kills six winners — one model harmlessly polling. Repeating is normal; repeating an error is the doomed signal.)

Telling a doomed loop to stop is robust; steering it to succeed is not. So the substrate detects, withholds compute when it is sure, and hands the rest to a human, rather than pretending it can fix what it can only see.

A.4 Where the substrate ends and a verifier begins

The failure that beats every detector here has no fingerprint at all: the agent runs cleanly, hits no error, contradicts nothing, and ends with a confident, polished, wrong summary, because it never re-read the state it claims to have changed. No byte already in the transcript reveals it; the only honest catch is to go re-read the world after the run. That is the job of the benchmark's own final judge, and the line this paper draws on purpose: an in-flight byte-clean substrate reads the evidence the environment already wrote; a final-state verifier re-reads the world to settle what no in-trace evidence can. The gap is not a flaw to hide — it is the boundary, where reading around the Vaseline runs out and going behind the mirror begins.
